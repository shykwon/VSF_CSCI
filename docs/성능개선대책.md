train loss 95.6은 단순히 느린 게 아니라 **발산하거나 완전히 막힌 상태**예요. 증상들을 보면 원인 체계가 명확합니다.

## 근본 원인 분석

```
S 행렬 초기화 = Identity (모든 변수 독립 가정)
        ↓
Wiener filter W가 의미 없는 값 출력
        ↓
V_miss_hat 스케일 폭발 → amplitude_mse 폭발
        ↓
L_spectral 폭발 → 전체 loss 폭발 → gradient 폭발
        ↓
S 행렬 학습 불가 → 악순환
```

ECG만 된 이유는 N이 작아서 이 악순환이 터지기 전에 수렴했기 때문입니다.

### 🔴 즉시 시도 — 학습 안정화 (1~2시간)

**1. amplitude_mse 폭발 차단 — V_miss_hat clamp**

가장 빠른 응급처치. V_miss_hat이 폭발하면 loss가 폭발하고 그게 다시 S를 망가뜨리는 악순환을 끊어줍니다.

비유하자면 "아무리 추정값이 이상해도 원본 신호 스케일을 넘지 못하게 제한"하는 것.

### 🟡 핵심 구조 개선 — S 행렬 초기화 (반나절)

**4. 데이터 기반 S 초기화** ← 가장 효과적

현재 Identity 초기화는 "아무 상관없다"는 가정에서 시작해 처음부터 엄청난 양의 학습을 요구합니다. 데이터에서 직접 추정한 값으로 시작하면 학습 부담이 극적으로 줄어듭니다.

```python
# main_csci.py Stage 2 시작 직전에 1회 호출
def init_S_from_data(csci_model, dataloader, device, n_batches=50):
    """
    학습 데이터 일부로 경험적 교차 스펙트럼 추정 → S 초기화
    비유: 시험 전에 족보 보는 것 — 처음부터 백지가 아닌 상태로 시작
    """
    F = csci_model.cs_estimator.F
    N = csci_model.cs_estimator.N
    S_accum = torch.zeros(F, N, N, dtype=torch.cfloat, device=device)
    count = 0
    
    with torch.no_grad():
        for i, (x, _) in enumerate(dataloader):
            if i >= n_batches:
                break
            x = x.to(device)  # [B, in_dim, N, T]
            # 신호 채널만 사용
            V = torch.fft.rfft(
                x[:, 0, :, :],  # [B, N, T]
                dim=-1, norm='ortho'
            ).permute(0, 2, 1)  # [B, F, N]
            
            # 배치 평균 교차 스펙트럼: S_ij = E[V_i * conj(V_j)]
            S_batch = torch.einsum('bfi,bfj->fij', V, V.conj()) / V.shape[0]
            S_accum += S_batch
            count += 1
    
    S_init = S_accum / count
    # 에르미트 대칭 강제
    S_init = (S_init + S_init.conj().transpose(-1,-2)) / 2
    
    csci_model.cs_estimator.S_real.data = S_init.real
    csci_model.cs_estimator.S_imag.data = S_init.imag
    print(f"S 초기화 완료: mean|S|={S_init.abs().mean():.4f}")
```

**5. Low-rank S 분해** ← 파라미터 수 문제 근본 해결

METR-LA 기준 현재 S 파라미터: `49 × 207² × 2 ≈ 4.2M`개. 이걸 짧은 Stage 2에서 처음부터 학습하는 건 무리입니다.

```python
# cross_spectral_estimator.py __init__ 수정
r = 16  # rank — N=207에서 16이면 파라미터 4.2M → 324K (1/13)

self.U_real = nn.Parameter(torch.randn(F, N, r) * 0.01)
self.U_imag = nn.Parameter(torch.zeros(F, N, r))
self.log_diag = nn.Parameter(torch.zeros(F, N))  # log 스케일로 양수 보장

def get_S(self):
    U = torch.complex(self.U_real, self.U_imag)          # [F, N, r]
    S_lr = U @ U.conj().transpose(-1, -2)                # [F, N, N] 에르미트 자동 보장
    S_diag = torch.diag_embed(F.softplus(self.log_diag)) # [F, N, N] 대각 양수 보장
    return S_lr + S_diag
```

비유: 207×207 행렬 전체를 외우는 대신, "공통 패턴 16개"만 학습하고 나머지는 조합으로 표현.

## 실험 순서 권장

```
Step 1 (오늘): lambda_reg=0.1 + V_miss_hat clamp
               → train loss가 안정되는지 확인
               
Step 2 (내일): 데이터 기반 S 초기화 추가
               → coherence가 올라가는지 확인
               
Step 3 (이번주): Low-rank S 분해 + Stage 1.5
               → METR-LA/SOLAR에서 VIDA 수준 접근 가능한지 확인
```

Step 1만으로 train loss 폭발이 잡히면 나머지는 성능 개선 문제이고, Step 1에서도 안 잡히면 **S 행렬 학습 경로 자체에 gradient가 흐르지 않는** 더 깊은 문제일 수 있습니다. 그 경우엔 `V_miss_hat.abs().mean()`을 매 epoch 출력해서 스케일 추이를 모니터링하는 게 먼저입니다.
